### Token Naming Convention

Follow the hierarchical token structure:

```
{category}.{property}.{variant}.{state}

Examples:
- color.button.primary.default
- color.button.primary.hover
- spacing.component.padding.sm
- typography.heading.large
```

## Component Naming

### Naming Guidelines

- Use **clear, descriptive names**: "Primary Button" not "Blue Button"
- Be **consistent**: If you use "Card Header", use "Card Content" (not "Card Body")
- Avoid **implementation details**: "Elevated Card" not "Card with Drop Shadow"
- Use **standard terminology**: "Disabled" not "Inactive" or "Greyed Out"

## Component Variants

### Creating Variants

Use Figma variants for:

1. **Visual styles**: Primary, Secondary, Outline
2. **Sizes**: Small, Medium, Large
3. **States**: Default, Hover, Focus, Active, Disabled
4. **Content types**: With icon, without icon

```
Button Component Variants:
- Variant: Primary, Secondary, Outline, Ghost, Link
- Size: Small, Medium, Large
- State: Default, Hover, Focus, Active, Disabled
- Icon: None, Left, Right, Only
```

### Variant Properties

Set variant properties systematically:

```
Property: Variant
Values: Primary, Secondary, Outline, Ghost, Destructive

Property: Size
Values: SM, MD, LG

Property: State
Values: Default, Hover, Focus, Active, Disabled

Property: Icon
Values: None, Left, Right, Icon-only
```

## Responsive Design

### Breakpoints

Design for these standard breakpoints:

- **Mobile**: 375px - 767px
- **Tablet**: 768px - 1023px
- **Desktop**: 1024px - 1439px
- **Large Desktop**: 1440px+

### Spacing Scale

Use the 4px/8px spacing system:

- **Micro**: 4px, 8px
- **Small**: 12px, 16px
- **Medium**: 20px, 24px
- **Large**: 32px, 40px, 48px
- **XLarge**: 64px, 80px, 96px

### Layout Patterns

```
Mobile-first approach:
1. Design mobile first (375px)
2. Adapt for tablet (768px)
3. Optimize for desktop (1024px+)
```

Annotate designs with:

## Design Handoff

### Preparing for Handoff

Before handing off to development:

1. **Clean up layers** - Remove unused components and layers
2. **Name everything** - Use clear, consistent naming
3. **Use tokens** - Replace hardcoded values with design tokens
4. **Add specs** - Annotate spacing, colors, typography
5. **Document states** - Show all interactive states
6. **Test contrast** - Verify accessibility
7. **Export assets** - Provide icons, images at correct sizes

### Handoff Checklist

- [ ] All layers properly named
- [ ] Design tokens used throughout
- [ ] Component variants are complete
- [ ] All states are designed (hover, focus, disabled)
- [ ] Spacing is annotated
- [ ] Typography specs are clear
- [ ] Colors use token names
- [ ] Accessibility requirements met
- [ ] Responsive behavior documented
- [ ] Assets exported (if needed)

## Asset Export Guidelines

### Icons

- **Format**: SVG
- **Size**: Design at 24Ã—24px
- **Stroke**: 2px (set as outline, not stroke)
- **Color**: Use `currentColor` for flexibility
- **Naming**: `icon-name.svg` (lowercase, kebab-case)

### Images

- **Format**: WebP (with PNG fallback)
- **Sizes**: Export @1x, @2x, @3x for mobile
- **Optimization**: Compress before committing
- **Naming**: Descriptive, kebab-case

### Illustrations

- **Format**: SVG for vectors, WebP for raster
- **Colors**: Use design token values
- **Accessibility**: Include alt text descriptions

## Review Process

### Design Review Checklist

Before requesting review:

- [ ] Follows design token system
- [ ] All variants and states included
- [ ] Meets accessibility standards
- [ ] Properly documented
- [ ] Consistent with existing components
- [ ] Responsive behavior defined
- [ ] Ready for development handoff

### Feedback Integration

When receiving feedback:

1. **Acknowledge** - Confirm you understand the feedback
2. **Ask questions** - Clarify any unclear points
3. **Update** - Make requested changes
4. **Document** - Note any decisions made
5. **Re-request review** - Once updates are complete
