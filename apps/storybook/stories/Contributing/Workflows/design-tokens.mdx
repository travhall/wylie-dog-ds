# Design Token Workflow

A comprehensive guide for updating and synchronizing design tokens between Figma and code.

## Overview

The Wylie Dog Design System uses a **token-based design approach** where visual properties (colors, spacing, typography, etc.) are defined in Figma and synchronized to code through the Token Bridge plugin. This ensures consistency and makes theme updates efficient.

### Token Flow

```
Figma Variables
      ↓
Token Bridge Plugin (apps/figma-plugin)
      ↓
packages/tokens/io/sync/
  ├── primitive.json   — raw OKLCH color scales, spacing values, etc.
  ├── semantic.json    — purpose-driven tokens referencing primitives
  └── components.json  — component-specific tokens referencing semantics
      ↓
Style Dictionary (pnpm --filter @wyliedog/tokens build)
      ↓
packages/tokens/dist/
  ├── tokens.css          — all variables in :root / .dark
  ├── semantic-light.css  — light mode semantic overrides
  ├── semantic-dark.css   — dark mode semantic overrides
  └── index.js / index.d.ts — TypeScript types
      ↓
React Components (via @wyliedog/ui/styles import)
```

> **Note:** The build pipeline uses [`@tokens-studio/sd-transforms`](https://github.com/Tokens-Studio/sd-transforms) as a Style Dictionary plugin to handle DTCG-format token files. This is a utility package, not the Tokens Studio Figma plugin — the Figma side is handled exclusively by the Token Bridge plugin.

## Token Architecture

### Token Hierarchy

Tokens are organized in three tiers:

**1. Primitive Tokens** — Raw values, context-free  
Location: `packages/tokens/io/sync/primitive.json`

```json
{
  "color": {
    "blue": {
      "600": {
        "$type": "color",
        "$value": "oklch(0.546 0.215 262.88)"
      }
    }
  }
}
```

**2. Semantic Tokens** — Purpose-driven, reference primitives  
Location: `packages/tokens/io/sync/semantic.json`

```json
{
  "color": {
    "interactive": {
      "primary": {
        "$type": "color",
        "$value": "{color.blue.600}"
      }
    }
  }
}
```

**3. Component Tokens** — Component-specific, reference semantic tokens  
Location: `packages/tokens/io/sync/components.json`

```json
{
  "color": {
    "button": {
      "default": {
        "background": {
          "$type": "color",
          "$value": "{color.interactive.primary}"
        }
      }
    }
  }
}
```

### Token Categories

The system includes tokens for:

- **Colors** — Brand, semantic, state colors
- **Typography** — Font families, sizes, weights, line heights
- **Spacing** — Margins, padding, gaps (numeric scale)
- **Border Radius** — Corner radius values
- **Shadow** — Box shadows and elevation
- **Motion** — Animation durations and easing curves
- **Opacity** — Alpha values

## Figma Setup

### Install the Token Bridge Plugin

1. Open Figma
2. Go to Plugins → Browse plugins
3. Search for **"Wylie Dog Token Bridge"**
4. Install and run the plugin

The Token Bridge plugin (`apps/figma-plugin`) is the only Figma integration for this design system. It reads Figma Variables and writes them to `packages/tokens/io/sync/*.json`, or the reverse — pushing JSON changes back to Figma Variables.

### Connect to the Repository

In the Token Bridge plugin settings, point the plugin at the repository's token files under `packages/tokens/io/sync/`. The plugin supports both pull (Figma → JSON) and push (JSON → Figma) sync directions.

## Making Token Updates

### Scenario 1: Update an Existing Token

**Example: Change the primary brand color**

1. **In Figma:**
   - Open the Token Bridge plugin
   - Navigate to the `color/interactive` Variable collection
   - Find `interactive/primary`
   - Update the value in OKLCH (e.g., `oklch(0.558 0.252 302.32)`)
   - Save the Variable change in Figma

2. **Export to Code:**
   - In Token Bridge, click **Export to JSON**
   - This writes the updated value to `packages/tokens/io/sync/semantic.json`

3. **Build CSS:**
   ```bash
   pnpm --filter @wyliedog/tokens build
   ```

   This runs Style Dictionary which:
   - Reads `io/sync/*.json`
   - Resolves all token references
   - Generates CSS variables in `dist/`

4. **Verify in Storybook:**
   ```bash
   pnpm dev:storybook
   ```

   Check that components reflect the new color across light and dark modes.

5. **Commit Changes:**
   ```bash
   git add packages/tokens/io/sync/semantic.json
   git add packages/tokens/dist/
   git commit -m "feat(tokens): update interactive primary to purple"
   ```

### Scenario 2: Add a New Token

**Example: Add a new spacing value**

1. **In Figma:**
   - Add a new Variable in the appropriate collection
   - Name it following the hierarchy convention (e.g., `spacing/layout/xl`)
   - Set the value

2. **Export and Build:**
   ```bash
   # After exporting via Token Bridge
   pnpm --filter @wyliedog/tokens build
   ```

3. **Use in Components:**
   ```tsx
   <div className="gap-(--space-layout-xl)">
     {/* Content */}
   </div>
   ```

4. **Update Documentation:**
   - Add the token to the relevant Storybook tokens story
   - Document the intended use case

### Scenario 3: Dark Mode Tokens

Dark mode is handled by overriding semantic tokens under the `.dark` class. Light and dark values are exported as separate CSS files.

The generated output looks like:

```css
/* dist/semantic-light.css */
:root {
  --color-background-primary: oklch(1.000 0.000 0.00);
  --color-text-primary: oklch(0.208 0.040 265.75);
}

/* dist/semantic-dark.css */
.dark {
  --color-background-primary: oklch(0.208 0.040 265.75);
  --color-text-primary: oklch(0.984 0.003 247.86);
}
```

To add or update dark mode values, update the Variable mode in Figma, export via Token Bridge, then rebuild.

## Token Naming Conventions

### Follow the Pattern

```
{category}.{property}.{variant}.{modifier}
```

### Examples

**Colors:**
```
color.background.primary
color.background.primary.hover
color.text.secondary
color.border.input.focus
color.interactive.primary
```

**Typography:**
```
typography.heading.xl.fontFamily
typography.heading.xl.fontSize
typography.heading.xl.lineHeight
typography.body.md.fontWeight
```

**Spacing:**
```
space.component.padding.sm
space.component.gap.md
space.layout.section.lg
```

### Naming Best Practices

✅ **Do:**
- Use semantic names: `color.background.primary` (not `color.blue.500`)
- Be specific: `space.button.padding.x` (not `spacing.4`)
- Follow hierarchy: primitive → semantic → component
- Match Figma Variable names exactly for consistent export/import

❌ **Don't:**
- Use implementation details: `shadow-drop-15px` (use `shadow.card.default`)
- Skip semantic layer: component tokens must reference semantic, not primitive tokens
- Use vague names: `spacing.big` (use `space.layout.section.lg`)

## Build Pipeline Details

### Configuration

The Style Dictionary configuration lives at `packages/tokens/style-dictionary.config.js`. It uses the `@tokens-studio/sd-transforms` transform group to handle DTCG `$type`/`$value` format, then applies project-specific transforms on top.

### Building Tokens

```bash
# Build from repo root (recommended)
pnpm --filter @wyliedog/tokens build

# Or from within the tokens package
cd packages/tokens
pnpm build
```

The build script runs in sequence:
1. `process-io` — pre-processes the sync JSON files
2. `style-dictionary` — generates CSS variables and manifest
3. `generate-font-loaders` — creates `@font-face` loader files
4. `generate-token-types` — creates TypeScript literal types

### Output Files

```
packages/tokens/dist/
├── tokens.css          # All variables (primitives + semantic light)
├── semantic-light.css  # Semantic layer, light mode (:root)
├── semantic-dark.css   # Semantic layer, dark mode (.dark)
├── index.js            # JS/TS exports
├── index.d.ts          # TypeScript types
├── css-vars.js         # CSS variable name constants
└── manifest.json       # Full token manifest
```

## Validation and Testing

### Token Validation

Before committing token changes, validate the token structure:

```bash
# Validate token references and structure
pnpm --filter @wyliedog/tokens test:tokens

# Generate a full token audit report
pnpm --filter @wyliedog/tokens audit

# Verbose audit with details
pnpm --filter @wyliedog/tokens audit:verbose
```

### Manual Testing Checklist

- [ ] All components render correctly in light mode
- [ ] All components render correctly in dark mode (`.dark` class on `<html>`)
- [ ] No broken token references (check browser console for missing CSS variables)
- [ ] Typography scales render correctly
- [ ] Spacing follows the defined scale
- [ ] Colors meet contrast requirements
- [ ] Animations use consistent timing

## Common Issues and Solutions

### Issue: Token Not Updating in Code

**Problem:** Changed a token in Figma but the component still shows the old value.

**Solution:**
1. Verify the change was exported from Token Bridge to `packages/tokens/io/sync/`
2. Run `pnpm --filter @wyliedog/tokens build` to regenerate CSS
3. Hard refresh the browser (Cmd/Ctrl + Shift + R)
4. Confirm `@wyliedog/ui/styles` is imported in your app's root layout

### Issue: Broken Token Reference

**Problem:** Token shows `undefined` in rendered output.

**Solution:**
1. Check that the referenced token exists in the JSON source:
   ```json
   // ❌ Typo in reference
   "$value": "{color.interactve.primary}"

   // ✅ Correct
   "$value": "{color.interactive.primary}"
   ```
2. Ensure token hierarchy is correct (component → semantic → primitive)
3. Re-run `pnpm --filter @wyliedog/tokens build`
4. Run `pnpm --filter @wyliedog/tokens test:tokens` to surface reference errors

### Issue: Dark Mode Not Working

**Problem:** Dark mode tokens not applying.

**Solution:**
1. Verify the `.dark` class is applied to `<html>`, not `<body>`
2. Confirm `@wyliedog/ui/styles` is imported — it bundles both light and dark CSS
3. Use browser DevTools to inspect computed values for `--color-background-primary`

### Issue: Build Fails After Token Changes

**Problem:** `pnpm --filter @wyliedog/tokens build` throws an error.

**Solution:**
1. Check the JSON syntax in the modified sync file
2. Run the audit to find structural issues:
   ```bash
   pnpm --filter @wyliedog/tokens audit:verbose
   ```
3. Ensure all `$value` references use the correct token path

## Best Practices

### 1. Always Use Tokens

```tsx
// ❌ Bad — Hardcoded values break theming and dark mode
<div style={{ color: '#3B82F6', padding: '16px' }}>

// ✅ Good — Design tokens via Tailwind v4 CSS variable syntax
<div className="text-(--color-text-primary) p-(--space-component-padding-md)">
```

### 2. Respect the Hierarchy

Components consume component tokens. Component tokens reference semantic tokens. Semantic tokens reference primitives. Never skip layers.

```tsx
// ❌ Bad — Component using a primitive token directly
<div className="bg-(--color-blue-600)">

// ✅ Good — Component using a component token
<div className="bg-(--color-button-default-background)">
```

### 3. Document Token Decisions

When adding or changing tokens, add a description in Figma (Variable description field) explaining why:

```
color/interactive/primary
Description: "Primary interactive color. Changed from blue to purple
per A/B test results (2024-01-15). References primitive color/purple/600."
```

### 4. Test Across Themes

Always test token changes in:
- Light mode
- Dark mode
- Reduced motion (check animation tokens)
